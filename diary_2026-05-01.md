# 2026-05-01 日記

今日は NextTrackAssist のコードを読んで、コンテナ起動からリクエスト処理までの流れをちゃんと理解しようとした。

---

## 今日わかったこと

### コンテナ起動の全体の流れ

Dockerfile の CMD がこれ。

```
python scripts/migrate.py && gunicorn -b 0.0.0.0:${PORT:-8000} "app:create_app()"
```

`&&` で繋いであるから、migrate.py が成功しないと gunicorn は起動しない。テーブルが壊れた状態でアプリが動き出すのを防ぐための設計。

---

### フェーズ①：migrate.py

まず `app/extensions.py` が読まれて、`engine`・`Base`・`SessionLocal` の3つが作られる。

次に `User` と `Track` モデルをインポートする。これをやることで、モデルの定義が `Base.metadata` に登録される。テーブルの設計図が Base に集まるイメージ。

そのあと `Base.metadata.create_all(bind=engine)` を呼ぶ。ここで初めてPostgreSQLに `CREATE TABLE` が飛ぶ。すでにテーブルがあれば何もしない。

最後に追加マイグレーション（002, 003）を順番に流して終了。これらは冪等なので何回実行しても安全。

---

### フェーズ②：gunicorn + create_app()

gunicorn が起動して `app:create_app()` を呼ぶ。`app/__init__.py` の `create_app()` 関数が動く。

中でやってることの順番：

1. logging 設定
2. `Flask(__name__)` でアプリ本体を生成
3. `app.config.from_object(Config)` で設定を注入（SECRET_KEY が未設定なら RuntimeError で止まる）
4. `csrf.init_app(app)` でCSRF保護を有効化
5. Blueprint を3本登録（auth / tracks / api）
6. エラーハンドラ（404, 500）を設定
7. `return app` → gunicorn に渡す

`create_app()` の中では `create_all()` を呼ばない。DB準備は migrate.py に一本化してある。関心の分離。

---

### フェーズ③：リクエスト待機 → 実際の処理

ポートにバインドしたら待機。あとはリクエスト駆動。

`/` にアクセスが来たら `tracks_bp` の `index()` が動く。

```python
db = SessionLocal()   # セッションを1本開く
try:
    tracks = db.query(Track).filter(Track.user_id == g.current_user.id).order_by(Track.id.desc()).all()
finally:
    db.close()        # エラーでも必ず閉じる
```

`query()` を組み立てる段階ではまだ SQL は飛ばない。`.all()` を呼んだ瞬間に SQL が確定して PostgreSQL に発行される。結果は `Track` クラスのインスタンスのリストに変換されて返ってくる。テンプレートでは `track.title` みたいにドットでアクセスできる。

---

## 今日の感想

起動フローって「とりあえず動く」だと全然わかってなかったけど、ファイルを1個ずつ追ってみたら「この行がこの処理をしてるんだ」ってちゃんと繋がった。migrate.py が先に走って DB を準備して、そのあとアプリが起動するっていう順番、ちゃんと理由があったのがよかった。

面接でも「起動フローを説明してください」って言われたら答えられる気がする。
