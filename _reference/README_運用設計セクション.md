## 運用設計

本番運用を見据えて、ログ・エラーハンドリング・障害対応の設計方針を整理しています。

### ログ設計

Python 標準の `logging` モジュールを使用し、アプリケーション全体で統一フォーマットのログを出力しています。

```
2026-04-26 15:30:00,123 [INFO] app.routes.tracks: トラック作成: id=42, user=1
2026-04-26 15:31:05,456 [ERROR] app.routes.tracks: CSVインポートに失敗 (user=1)
Traceback (most recent call last):
  ...
```

| レベル | 用途 | 例 |
|--------|------|-----|
| INFO | 正常な操作の記録 | ログイン成功、トラック作成、CSVインポート完了 |
| WARNING | 異常だがサービスは継続 | ログイン失敗（パスワード不一致）、重複メール登録 |
| ERROR | エラー発生 | DB書き込み失敗、500エラー |

本番環境では Railway のログビューアで確認可能。将来的には Datadog 等の監視サービスに連携し、ERROR レベルのログ発生時に Slack 通知を送る構成を想定しています。

### エラーハンドリング方針

DB操作は全て `try / except / finally` で囲み、例外時は `rollback()` + `logger.exception()` でスタックトレースを記録します。

| 障害パターン | 影響 | 対処 |
|-------------|------|------|
| DB接続断 | 全機能停止 | SQLAlchemy の `pool_pre_ping=True` で接続確認。復旧まで 500 エラーページを表示 |
| 不正なCSVアップロード | インポート機能のみ | バリデーションで弾き、エラー理由をユーザーに表示。ログに WARNING 記録 |
| 不正なリクエスト（CSRF / バリデーション違反） | 該当リクエストのみ | Flask-WTF が 400 を返却。ログに WARNING 記録 |
| 想定外の例外 | 該当リクエストのみ | カスタム 500 エラーページを表示。`logger.exception()` で詳細を記録 |

### SLO（Service Level Objective）

本番運用時の品質目標として、以下の SLO を設定しています。

| 指標 | 目標値 | 計測方法（想定） |
|------|--------|----------------|
| API応答時間 | 95パーセンタイルで 200ms 以内 | アクセスログの集計 |
| 稼働率 | 99.5%（月間ダウンタイム約3.6時間以内） | 外形監視（UptimeRobot 等） |
| エラー率 | 5xx レスポンスが全リクエストの 0.5% 以下 | ログ集計 |

現時点では Railway のログとダッシュボードで手動監視しています。ユーザー数が増加した段階で、外形監視ツールや APM（Application Performance Monitoring）の導入を予定しています。
